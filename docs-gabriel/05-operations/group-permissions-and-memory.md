# 채팅 그룹별 호스트 접근 권한 제어 및 CLAUDE.md 설정 가이드

NanoClaw 시스템은 개별 사용자와의 1:1 채팅이나, 서로 다른 단톡방(WhatsApp/Telegram) 환경을 제각기 **격리된 그룹 컨테이너(Group Container)** 로 분리하여 실행합니다. 

이 문서에서는 각 채팅 그룹방마다 외부 호스트 디렉터리 접근 권한을 어떻게 차등 부여하는지, 그리고 에이전트의 성격과 기억을 통제하는 `CLAUDE.md`가 계층적으로 어떻게 적용되는지 설명합니다.

---

## 1. 그룹별 호스트 디렉터리 접근 권한 제어 (Mounts)

각 대화방의 에이전트들은 기본적으로 자신의 작업 샌드박스 폴더(`groups/{folder}/`) 외에는 호스트 PC의 어떤 데이터도 볼 수 없습니다. 특정 그룹에게만 호스트의 외부 폴더 접근 권한을 열어주려면 **2개의 관문**을 통과해야 적용됩니다.

### 관문 A: 호스트 전역 보안 정책 (`mount-allowlist.json`)
아무리 각 방의 에이전트가 권한을 요구해도, 호스트가 허락하지 않은 루트 경로는 절대 마운트되지 않습니다. 호스트 데몬을 통제하는 설정 파일에서 먼저 전역 범위를 지정합니다.

**파일 위치:** `~/.config/nanoclaw/mount-allowlist.json`
```json
{
  "allowedRoots": [
    {
      "path": "~/obsidian",
      "allowReadWrite": true,
      "description": "업무/개인 노트 보관소 (접근 허용 구역)"
    }
  ],
  "blockedPatterns": ["secret", ".ssh"],
  "nonMainReadOnly": true
}
```
*   `nonMainReadOnly`의 값이 `true`이면, 메리트가 높은 관리자 방(main)을 제외한 모든 외부 일반 그룹방은 **파일 내용을 읽을 수만 있고 훼손할(수정/삭제) 수 없도록 강제 읽기 전용**으로 묶어버립니다.

### 관문 B: 그룹별 채널 맞춤 배정 (`registered_groups` DB)
위의 전역 관문을 통과한 경로 중에서, 특정 A그룹에게는 A폴더만 통제하여 컨테이너로 연결해 줄 수 있습니다. 이를 설정하는 방법은 크게 두 가지가 있습니다.

#### 방법 1: 메인(관리자) 그룹의 Claude에게 대화로 요청하기 (가장 쉬운 방법)
`main` 방의 에이전트는 호스트 DB에 접근하여 다른 그룹들의 환경을 세팅할 수 있는 특권 도구를 가지고 있습니다. `main` 방에서 다음과 같이 요청하세요.

> **사용자:** "@Andy, `group-A` 방에 호스트의 `~/obsidian/group-A` 폴더를 컨테이너 경로 `notes`로 마운트할 수 있도록 설정해 줘."

이렇게 하면 에이전트가 알아서 SQLite에 레코드를 업데이트해 줍니다.

#### 방법 2: 호스트 터미널에서 직접 SQLite DB 수정하기
직접 쿼리를 넣어서 설정할 수도 있습니다. 호스트 PC의 터미널에서 아래 명령어를 실행하여 등록된 그룹의 `container_config` 속성을 업데이트합니다.

```bash
# SQLite DB 직접 수정 쿼리
sqlite3 store/messages.db "
UPDATE registered_groups
SET container_config = json('{
  \"additionalMounts\": [
    {
      \"hostPath\": \"~/obsidian/group-A\",
      \"containerPath\": \"notes\",
      \"readonly\": false
    }
  ]
}')
WHERE folder = 'group-A';
"
```

설정 후 **해당 그룹의 작동 중인 컨테이너가 있다면 강제로 종료(`docker stop`)** 시켜야 합니다. 다음번 메시지가 도착하여 새 컨테이너가 열릴 때, 오직 `group-A` 방의 에이전트만 컨테이너 내부의 `/workspace/extra/notes/` 경로에서 해당 외부 파일들을 열람하고 상호작용할 수 있게 됩니다.

## 2. 에이전트의 기억과 뇌구조 (`CLAUDE.md` 계층 시스템)

NanoClaw의 에이전트들은 사용자와 긴 세월 대화하면서 **핵심 규칙과 맥락적인 장기 기억력**을 파일 형태로 저장하며 관리합니다. 그 뇌의 역할을 하는 핵심이 바로 `CLAUDE.md` 파일입니다. 

NanoClaw는 여러 겹의 메모장(MD)을 동시에 겹쳐서 읽어들이며 에이전트의 자아를 형성합니다.

### 🌎 Layer 1: 공통 전역 룰셋 (`groups/global/CLAUDE.md`)
*   **파일 위치**: 호스트 PC의 `nanoclaw/groups/global/CLAUDE.md`
*   **접근 권한**: 
    - 모든 방(main, group-A, group-B)의 비서들이 이 파일을 동시에 읽어 들입니다.
    - 단, 이 파일 자체를 터미널로 수정하거나 내용을 고칠 수 있는 권한(쓰기 권한)은 오직 **메인(main) 1:1 방의 컨테이너**에만 존재합니다.
*   **활용법**: 봇의 기본 성격(당신의 이름은 Andy입니다), 항상 한국어로 대답할 것, 욕설 금지 등 **어느 대화방에서 활동하건 공통적으로 준수해야 할 뼈대 헌법**을 작성해두는 곳입니다.

### 🏠 Layer 2: 채팅방(그룹) 전용 메모리 (`groups/{그룹}/CLAUDE.md`)
*   **파일 위치**: 예) `nanoclaw/groups/main/CLAUDE.md` 또는 `groups/group-B/CLAUDE.md`
*   **접근 권한**: 
    - 타 그룹은 절대 읽을 수 없고, 오롯이 해당 방에 할당된 컨테이너만 이 파일을 읽고 자유롭게 기록할 수 있습니다.
*   **활용법**: 
    - 해당 단톡방 구성원들의 이름, 방의 목적, 진행 중인 장기 프로젝트, 해당 사용자들과의 이전 뉘앙스 등 **그 방 안에서만 적용되는 맥락들**을 AI가 스스로 끄적이고 참조합니다.
    - 컨텍스트 오버플로우(대화 한계 토큰 초과)가 발생해서 수개월의 과거 대화가 압축(Compression)될 때도 핵심 기조가 이 파일에 저장되어 보존됩니다.

### 🗂️ Layer 3: 외부 마운트 프로젝트 메모장 (Additional Mounts)
에이전트는 자신에게 추가 마운트된 디렉터리(`workspace/extra/notes`) 안으로 들어가 파일을 작업할 때도 본능적으로 추가 룰을 찾습니다.
*   만일 추가로 떼어준 `~/obsidian/group-A/` 폴더 최상위에 또 অন্য `CLAUDE.md` 파일이 존재할 경우, 에이전트는 이 작업 디렉터리에 상주할 때 해당 폴더에 명시된 규칙(예: 파일 저장 포맷, 문서 작성 기법 매뉴얼)까지 복합적으로 함께 읽어들여 적용합니다.

---

위의 3가지 계층 구조 방식을 적절히 설계하면, 여러 채팅방과 여러 프로젝트의 묶음 속에서도 에이전트가 어떤 방에서는 관리자로써 시스템 전역을 관장하고, 어떤 단톡방에서는 특정 노트 파일 내용만 제한적으로 다루는 일반 비서로 완벽하고 안전하게 롤플레잉(Role-Playing)할 수 있습니다.
